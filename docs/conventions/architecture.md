# アーキテクチャ規約

このファイルは、モジュール境界の原則と、それを機械強制する仕組みの「なぜ」を説明する人間向けの正本である。ルール自体は `make arch`(dependency-cruiser / import-linter)が CI でブロッキング強制する（本テンプレートの根拠は [tech-radar.md](../governance/tech-radar.md) の「依存境界lint」= Adopt / W4.10）。散文で守らせるのではなく、この文書は判断の背景だけを担う。

## 中核原則: 境界は宣言し、方向を固定し、lintで守る

1. **レイヤを宣言する** — どのフォルダがどの層かを設定ファイルに明示する（暗黙の慣習に頼らない）。
2. **依存方向を一方通行にする** — 上位層は下位層を呼べるが、下位層は上位層を知らない。逆流と循環は設計の腐敗の初期兆候。
3. **lintで強制する** — 宣言した方向を dependency-cruiser（TS）/ import-linter（Py）が検査し、違反はCIで止める。人間のレビュー注意力に依存しない。

この3点で「スパゲッティ化」の主要因である双方向依存・循環・層飛ばしを機械的に排除する。エージェントは境界を目視で守れない前提に立ち、境界を実行可能なルールに落とす。

## モジュールが「隠すもの」を書く（Parnas 情報隠蔽）

Parnas (1972) の情報隠蔽が示すのは、モジュール分割の正しい軸は「処理のステップ」ではなく「将来変わりうる決定」だという点である。よって規約はこうする。

- 各モジュール（パッケージ / features 単位 / サービス）の入口に、**「このモジュールは何を隠蔽するか（将来変わりうる決定は何か）」を一言添える**。例: 「永続化の実体（今はSQLite、将来別DBもありうる）を隠す」「外部SEO APIのレスポンス形状の揺れを吸収する」。
- 変わりやすい決定ほど境界の内側に閉じ込め、公開インターフェースは安定させる。
- この一言があると、次に触るエージェント/人間が「どこを触ってよく、何を壊してはいけないか」を推測せずに判断できる。

## 既定のレイヤ構成（プロファイル別）

境界の具体形はプロファイルが持ち込む。素のテンプレートでは `make arch` は no-op で、プロファイル導入（`tools/bootstrap.sh`）で有効化される。

### TypeScript プロファイル

`app → features → core` の一方向依存。定義は `profiles/typescript/.dependency-cruiser.cjs` にある。

- `app`: エントリ・組み立て層。`features` と `core` に依存してよい。
- `features`: 機能単位。`core` に依存してよいが、他の `features` や `app` を知らない（機能間の横結合を禁止）。
- `shared/core`: 汎用基盤。他層を知らない（最下層）。
- **循環禁止**（相互import）と **orphan（どこからも参照されない孤立ファイル）警告** を併せて有効化する。

### Python プロファイル

`api → services → models` の一方向。import-linter の layers contract として `profiles/python/pyproject.toml` に定義。

- `api`: 入口（HTTP / CLI ハンドラ）。`services` を呼ぶ。
- `services`: ドメインロジック。`models` を使う。
- `models`: データ構造・永続化。上位を知らない。

## 既存コードへの後付け: freeze / baseline 方式

境界lintは新規プロジェクトだけのものではない。乱れた既存コードにも段階導入できることが汎用性の要で、Netflix の大規模運用（ArchUnit 358ルールを約5,000リポジトリへ展開）や Adidas の事例が示すのは「一括修正を強制せず、悪化だけを止める」導入が現実的だという点である。手順は次の通り。

1. **現状違反を測る** — 導入時点の全違反を洗い出す。
2. **baseline として凍結する** — 既知の違反リストを baseline ファイルに固定し、CIはこの既知分を素通しする。
3. **新規違反のみ止める** — baseline に無い新しい違反が入ったらCIで落とす（悪化の防止）。
4. **baseline を徐々に減らす** — リファクタのたびに既知違反を1件ずつ消し、baseline を空へ近づける。減ることはあっても増えないように運用する。

これにより「最初から完璧な境界」を要求せず、`tools/adopt.sh` で後付け導入する既存リポジトリでも今日から回せる。

## `make arch` との関係

- `make arch` はプロファイルの `Makefile.profile` が実装し、TSなら dependency-cruiser、Pyなら import-linter を実行する。
- `make check` はこれを含む全ゲートをまとめて走らせ、CI（`.github/workflows/ci.yml`）が必須チェックにする。
- したがって本文書のルールは「読んで守る」ものではなく「破ると赤くなる」もの。逸脱を既定に変えたい場合は [rubric.md](../governance/rubric.md) の判定（intake）を経る。
