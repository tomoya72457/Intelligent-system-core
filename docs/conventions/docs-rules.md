# ドキュメント規約

このファイルは、`tools/check_structure.py` が機械強制するドキュメント規約の「なぜ」を説明する人間向けの正本である。ルールの実体はチェッカのコードにあり、`make structure`(および `make check`・`tools/githooks/pre-commit`・CI)で常時検証される。規約を変えたいときは、この文書ではなくチェッカの変更として提案する(散文より機械強制)。

## 1. 常時コンテキストの予算

- **AGENTS.md**: ソフト150行(警告)/ ハード200行・24,576バイト(エラー)。
- **CLAUDE.md**: 1行目に `@AGENTS.md` を含むこと+全体800バイト以下。ポインタ以外の内容を持たせない。
- なぜ: 常時読み込まれる文書は1行ごとにエージェントの注意を消費する。長文脈の中間は読み飛ばされ(Lost in the Middle)、命令が増えるほど遵守率が落ち(IFScale)、Codex は 32KiB で打ち切る。「書けば読まれる」は成り立たないので、予算で総量を固定し、詳細は下の層へ逃がす。
- 逃がし先の3層: 常時規範(`AGENTS.md`)→ 作業手順(`.claude/skills/`)→ 参照知識(`docs/`)。下の層ほど「必要な時だけ」読まれる(段階開示)。

## 2. ルータ README(用途起点の道案内)

- `docs/README.md` と `profiles/README.md` は必須(チェッカが存在を検証)。
- ルータはファイル一覧の羅列ではなく「〜したい → このファイル」の用途起点の表で書く。エージェントは目的を持って検索するため、用途→場所の写像が最短経路になる。
- 新しいディレクトリ・文書を足したら、対応するルータに1行追加する。ルータに載らない文書は存在しないのと同じ(発見されない)。

## 3. 追記型台帳と「全文再生成禁止」

- `docs/adr/INDEX.md`・`docs/governance/tech-radar.md`・`docs/governance/intake/` は**追記型台帳**である。更新は行の追加・当該行の修正で行う。
- **LLM による台帳の全文再生成は禁止**。LLM に台帳全体を書き直させると、既存行の要約・省略・言い換えが混入し、蓄積した事実が静かに失われる(context collapse)。台帳の価値は「過去の判断が原文のまま残る」ことにある。
- 生成し直してよいのは新規追加分のみ。既存行に触るのは、その行の事実が変わった時(状態変更・置換)だけ。

## 4. stale 参照と物理再編のプリフライト

- 文書内のリポジトリ内パス参照は実在するファイルを指すこと。チェッカは設定された旧パスパターン(初期は空)を stale 参照として検出する。
- **物理再編(ファイル移動・改名)のプリフライト**: 移動・改名の前に、旧パスを参照している文書を `grep` で洗い出し、参照の更新を同じ変更に含める。移動後に旧パスパターンをチェッカの stale リストへ追加すると、以後の逆戻り・貼り忘れを機械検出できる(ミスの規約化)。
- なぜ: エージェントは文書内のパスをそのまま辿る。壊れた参照は人間より高い確率で誤動作(存在しないファイルの新規作成等)につながる。

## 5. ルート直下の統制

- リポジトリのルート直下に置けるファイルは許可リスト方式(チェッカが検証)。散らかりの初期兆候である「とりあえずルートに置く」を止める。
- 一時ファイルは `scratch/`(gitignore 済)へ。恒久的な文書は `docs/` の適切な役割フォルダへ。

## 6. 生きた文書の運用(棚卸し)

- **ミスは2回目で規約化**: 同じ誤りを2回観測したら、規約・hooks・チェッカへ追記して機械強制に落とす(`.claude/skills/incident/SKILL.md` の手順)。1回目は偶然でも2回目はパターン。
- **削除テスト**: 規約行を見たら「この行を消したらエージェントはミスするか?」と自問する。ミスしないなら消す。守られない規約が並ぶほど、守るべき規約の遵守率が下がる。
- **月次棚卸し**: 月に一度、AGENTS.md・skills・本規約群を見て、使われていない記述を削除する。文書は増やすより減らす方が難しく、定期的な強制がないと必ず太る。
