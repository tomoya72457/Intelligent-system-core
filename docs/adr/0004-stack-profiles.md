# ADR-0004: スタックは typescript / python / docs の3プロファイル

このファイルは、テンプレートが同梱する技術スタック構成の確定判断(2026-07-09 壁打ち)を記録する。

- **日付**: 2026-07-09
- **状態**: 承認

## 文脈

想定プロジェクト(SEO ダッシュボード・personal-OS・議事録ツール等)は Web アプリ、データ処理、ナレッジベースに大別される。テンプレートが単一スタックを固定すると用途が狭まり、逆に無限定にすると `make test` / `make lint` 等の検証契約(原則3)が空文になる。各スタックの lint・型・境界チェックのツール選定と設定は毎回ゼロから作ると重く、規約のばらつきも生む。

## 決定

**typescript / python / docs** の3プロファイルを `profiles/` に同梱する。各プロファイルは `PROFILE.md` のマニフェストを持ち、`tools/bootstrap.sh` が選択されたプロファイルを展開して `make test` / `make lint` / `make typecheck` / `make arch` の実体を与える(未導入時は no-op+案内表示)。展開後 `profiles/` は削除される。

- **typescript**: vitest+biome+tsc+dependency-cruiser(層: app→features→shared/core)
- **python**: uv+ruff+mypy+pytest+import-linter(層: api→services→models)
- **docs**: コード無しナレッジリポジトリ用(構造検証のみ強化)

## 検討した代替案(必須)

- **案B: 単一スタック(TypeScript のみ)固定** — 却下。データ処理系は Python が実態で、ナレッジリポジトリにはコードスタック自体が不要。単一固定は「多様なシステムの出発点」という目的(ADR-0001 の文脈)に反する。
- **案C: プロファイル無し(検証契約だけ定義し、中身は毎回手組み)** — 却下。`make check` が最初から動かない出発点は検証ループ(原則3)を最初の数日欠落させ、その間にエージェントの品質が守られない。ツール選定の反復コストも毎回発生する。
- **案D: 5以上のプロファイル(Go・Rust 等も)** — 却下。現時点で実需がなく(YAGNI)、各プロファイルは設定・例・文書の維持コストを持つ。必要になったらプロファイル追加は可逆・増分で可能(two-way door)。

## 結果

- どの想定プロジェクトでも、bootstrap 直後から `make check` が実体を持つ。
- Makefile の検証契約(§ コマンド表)は3プロファイルで共通のまま、実装だけが差し替わる。依存境界 lint(`docs/governance/intake/2026-07-09-architecture-fitness-functions.md`)も各プロファイルに既定の層構成として入る。
- トレードオフ: 3プロファイル分の設定維持が必要。各プロファイルを最小(設定+最小例+1ページ規約)に保って緩和する。

## Confirmation(実装後に追記)

`profiles/` 配下に3プロファイルのマニフェストと設定一式が存在し、JSON/TOML/YAML の構文検証を生成時に実施。bootstrap によるプロファイル展開の実地確認は、派生リポジトリでの初回実行時に本欄へ追記する。

## 信頼度と再検討トリガー

- **信頼度**: 中(ツール選定は各エコシステムの主流に沿うが、組み合わせの実運用実証はこれから)
- **再検討トリガー**: 想定プロジェクトに3プロファイルで賄えない実需(モバイル・別言語等)が出たら。選定ツールの主流交代(例: biome/ruff の後継)が起きたら。
