# adoption-judge ルーブリック(正式版)

このファイルは、テンプレート/プロジェクトの「既定(デフォルト)」を変える提案を評価するための判定基準の正本である。`.claude/agents/adoption-judge.md`(判定を実行するサブエージェント)と `.claude/skills/intake/SKILL.md`(判定を起動するスキル)は、本ファイルを読み込んで適用する。判定結果は `docs/governance/intake/` に記録し、`docs/governance/tech-radar.md` の台帳に1行として反映する。

## 対象と非対象

- **対象**: 「既定」を変える提案 — 新ツール・新規約・新ワークフロー・新エージェント機能を、テンプレートまたはプロジェクトの標準装備にすべきかどうかの判断。
- **非対象**: 日常の実装判断(この関数をどう分割するか、この変数名をどうするか等)には**適用しない**。ルーブリックの過剰適用は、判断のたびに儀式を挟む「複雑性コスト」を自ら生み、原則10の趣旨に反する。

## プロセス(7段階)

1. **適用可能性ゲート**: 環境制約で物理的に使用不可なもの(例: 個人GitHubアカウントで使えない機能、Freeプランに存在しない機能)は、点数を付けるまでもなく即 **Reject**。ただし「なぜ不可か」は記録に残し、蒸し返しを防ぐ。
2. **可逆性判定(two-way door)**: 撤去のコストで分類する。**可逆** = 設定1つ・ファイル数枚の削除で元に戻せる。**不可逆** = データ・履歴・外部契約(公開API・リリース済みバージョン等)に波及し、後戻りに追加作業や外部調整を要する。不可逆な提案は Trial 以上にするなら書面根拠を必須とし、Adopt には実運用での実証を必須とする。
3. **両論生成**: 賛成論・反対論を**それぞれ3点以上**、各点に根拠を添えて明示的に書く。これは「一度採用に傾くと反証を探さなくなる」単一パスバイアス(self-preference bias)への構造的対策であり、省略してはならない。
4. **5軸採点**: 下表の各軸を0〜5点で採点し、重み付き平均を出す。
5. **判定**: 加重平均を閾値に照らす(後述)。
6. **拒否権(veto)**: いずれかの軸が1点以下なら Adopt 不可(上限 Trial)。セキュリティを能動的に悪化させる提案は、点数に関わらず Reject できる。
7. **記録**: `docs/governance/intake/YYYY-MM-DD-<slug>.md` に判定記録を作り、`docs/governance/tech-radar.md` の表に1行追加する。Adopt 項目は12ヶ月TTLで再審査対象。

## 5軸と重み

| 軸 | 重み | 5点の基準 | 1点の基準 |
|---|---|---|---|
| E: エビデンス強度 | 0.30 | 査読/大規模データが直接支持、または独立3つ以上+エコシステムの収斂 | 単発の逸話・未検証の数値 |
| G: 汎用性 | 0.20 | 3プロファイル(typescript / python / docs)すべてに効く | 特定プロジェクト固有 |
| A: エージェント効果 | 0.25 | 検証可能性・決定性・コンテキスト経済に直接寄与 | エージェント作業にほぼ無関係 |
| M: 維持コスト(逆) | 0.15 | 置けば腐らない静的ファイル | 常時手入れが要り腐敗しやすい |
| C: 複雑性コスト(逆) | 0.10 | 利用者が意識しなくてよい | 毎回の作業に儀式を追加する |

M・C は「逆」軸である。維持・複雑性のコストが**低い**ほど高得点になる。

## 判定閾値

| 加重平均 W | 判定 | 意味 |
|---|---|---|
| W ≥ 4.0 | **Adopt** | 既定として装備する |
| 3.0 ≤ W < 4.0 | **Trial** | 同梱するがオプトイン。TTL 90日。実運用の実証なしに Adopt へ昇格させない |
| 2.0 ≤ W < 3.0 | **Assess** | 記録のみ。同梱しない |
| W < 2.0 | **Reject** | 採用しない(記録は残す) |

## 拒否権(veto)

- **軸の下限**: E/G/A/M/C のいずれかが1点以下なら、加重平均が高くても Adopt にはできない(上限 Trial)。弱点が1つでも致命的な軸は、平均で薄めて見過ごさない。
- **セキュリティ悪化**: 攻撃面を能動的に広げる提案は、他の軸が高くても Reject にできる(原則8)。

## 記録とTTL

- 各判定は `docs/governance/intake/YYYY-MM-DD-<slug>.md` に、両論・採点・判定・根拠ソース(URL)込みで残す。Reject も残す(同じ提案の蒸し返しを1回で止めるため)。
- `docs/governance/tech-radar.md` はこれらの判定の一覧台帳。intake 記録1件がテーマ単位で複数行を持つこともある。
- **Adopt** 項目は採用から12ヶ月後に再審査(前提の陳腐化を検出するため)。**Trial** 項目は90日TTLで、期限までに実運用の実証がなければ Assess/Reject へ落とす。

## 運用手順(どう起動するか)

1. 新手法・新ツールを既定に入れたくなったら `.claude/skills/intake/SKILL.md` のスキルを起動する。
2. スキルは `.claude/agents/adoption-judge.md` サブエージェントを呼ぶ。サブエージェントは本ルーブリックを読み込み、過去記録(`docs/governance/tech-radar.md` と `docs/governance/intake/`)を grep して、既に Reject 済み・判定済みでないかを先に確認する。
3. 両論 → 採点表 → 判定 → intake 記録の Markdown を出力する。
4. 人間が記録を確認し、`docs/governance/tech-radar.md` へ反映してコミットする。既定変更は原則8によりセキュリティレビュー対象であり、`docs/adr/` へのADR記録を伴うことがある。
