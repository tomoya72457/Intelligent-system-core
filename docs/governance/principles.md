# 設計原則10ヶ条

本テンプレートの全構成要素はこの10原則に照らして採否を判定している(判定手順は `rubric.md`、結果台帳は `tech-radar.md`)。カッコ内は主な根拠の種別。

1. **散文より機械強制** — 守らせたいルールは hooks / CI / linter に落とす。文書は「なぜ」を説明する場(日本語圏7独立ソース+各社公式ドキュメントの収斂)
2. **常時コンテキスト最小・段階開示** — 正本 AGENTS.md はソフト150行/ハード200行・24KiB。詳細は skills / docs へ逃がす(査読研究3本: Lost in the Middle / Context Rot / IFScale+Codex 32KiB 切り捨て実測)
3. **テスト=検証ループ** — `make check` 1コマンドで全ゲート実行。CI で必須チェック化(SWE-bench 系研究+全ベンダー第1推奨)
4. **小バッチ** — 1 PR = 1論理変更・目安±400行。超過は CI が警告(DORA 2024/2025+Google小CL規範+24,014エージェントPR実証)
5. **境界の適応度関数** — 依存方向 lint(dependency-cruiser / import-linter)を CI ゲート化(Parnas 1972+Netflix 5,000リポジトリ実績)
6. **二層強制** — ローカル githooks+CI バックストップ。「ローカルのみ=再クローンで消える」を許さない(参考3リポジトリ共通の弱点への対策)
7. **AIレビュー=助言、ブロック=決定的チェックのみ** — 判定操作攻撃の成功率89-99%(査読研究)+ベンダー自身の非ブロッキング設計
8. **エージェント設定=セキュリティ境界** — settings / hooks / workflows / githooks の自己編集を遮断(設定ファイル注入が最重大ベクターとする研究)
9. **本番書込は「PRマージ=承認」** — 提案と適用の workflow 分離+concurrency 直列化+書込直前の hash 再確認(RFC 9110+実事故2件の教訓)
10. **判定なき既定変更の禁止** — 新手法・新ツールの既定化は adoption-judge の判定記録を経る(本ガバナンス機構そのもの)

## 運用

- 「原則N」の番号は本ファイルが定義元。既存番号は変えない(追加は11以降に追記)。
- 原則の変更・追加は ADR を起こし、`tech-radar.md` の関連判定を再確認する。
