# プレイブック: タスク分解

このファイルの目的: 大きな要望を、コーディングエージェントが着手・検証できる「小タスク」に分解する手順をまとめる。1タスク=1PR=1論理変更を守るための実務ガイド。

## なぜ小さく切るか

スコープが明確なタスクほど、エージェントの生成物はマージまで到達しやすい(24,014件のエージェントPR分析でスコープ明確性がマージ可否をAUC 0.72で予測、arxiv:2512.21426)。小バッチはDORA/Googleの調査でAI支援開発の効果を増幅する条件の一つに挙がる一方、AI採用時に安定性が下がりやすいという逆風もあり、変更を小さく保つ規律で相殺する。分解は「エージェントを賢くする」より前に効く、最も費用対効果の高い介入。

## 標準フロー

1. **起票**: `.github/ISSUE_TEMPLATE/task.yml` のIssue Formで起票する。目的・受け入れ基準は必須。自由記述の空Issueは使わない(構造化フォームへ誘導済み)。
2. **受け入れ基準を検証可能な形式で書く**: 「きれいにする」「速くする」のような形容詞で終えない。観測可能な条件に落とす。
   - 良い例: 「`make check` が緑」「`/health` に GET すると 200 と `{"status":"ok"}` が返る」「一覧画面で削除ボタンを押すと行が消え、リロード後も消えている」。
   - 悪い例: 「使いやすくする」「パフォーマンス改善」。何をもって完了かをエージェントも人間も判定できない。
3. **1タスク=1PR=±400行以内**: 変更が±400行を超えると `.github/workflows/ci.yml` が `::warning` を出す(ブロックはしない)。警告が出たら「本当に1論理変更か」を疑い、分割を検討する。
4. **ラベルで状態を示す**: 受け入れ基準まで揃い着手可能なら `agent:ready`。仕様が曖昧で先に仕様化が必要なら `needs-spec`。人間の判断待ちなら `human-review`。他タスク待ちなら `blocked`。
5. **着手**: `agent:ready` のIssueをエージェントに渡す。エージェントは受け入れ基準を「検証コマンド」に翻訳して実装し、`make check` で自己検証する。

## 分解のコツ

- **縦に切る**: 「1つの検証可能な振る舞い」で切る。UI・API・DB をまたいでも、エンドツーエンドで観測できる薄い一本にする。層ごとの横切り(「まずDB全部」等)は検証を後ろ倒しにし、統合で破綻しやすい。
- **依存を減らす**: タスク間の前提を最小化する。Aが終わらないとBが検証できない連鎖は、レビューとロールバックのコストを跳ね上げる。
- **スコープ外を明記する**: `task.yml` の「スコープ外」欄に、やらないことを書く。エージェントの依頼範囲逸脱を防ぐ最も効く一文。
- **非自明なタスクはスペック駆動へ**: 3ファイル超・新規サブシステム・外部API契約変更のいずれかに触れるなら、直接実装せず `.claude/skills/spec/SKILL.md` のスペック駆動フロー(Trial)で requirements/design/tasks に分解してから着手する。閾値未満は直接実装でよい(儀式を全タスクに課さない)。

## 分解の例

要望「ユーザーがプロフィール画像をアップロードできるようにする」を分解する。

- 悪い分解 — 1タスクのまま渡す: UI・アップロードAPI・ストレージ・バリデーション・表示が全部入りで±400行を超え、検証が「全部できたか」の一括判定になる。
- 良い分解 — 縦切りで3タスクに分ける:
  1. `feat: 画像アップロードのAPIエンドポイントを追加` — 受け入れ基準「`POST /api/avatar` に画像を送ると 200 と保存先URLが返り、非画像は 415 を返す」。
  2. `feat: プロフィール画面にアップロードUIを追加` — 受け入れ基準「画面からファイルを選ぶと 1 のAPIが呼ばれ、成功時にプレビューが差し替わる」。
  3. `feat: 保存済みアバターを一覧・詳細に表示` — 受け入れ基準「アバター設定済みユーザーの行に画像が出る/未設定は既定アイコン」。

各タスクが単独でマージ・ロールバックでき、受け入れ基準がそのまま検証コマンド/操作手順になっている。1と2の間には依存があるが、1が先に検証可能な形で閉じているため連鎖が浅い。

## PRとの接続

分解したタスクは1つのPRに対応する。PR作成時は `.github/pull_request_template.md` のチェックに答える:

- 変更は1論理単位・±400行以内か。
- 検証コマンドと結果を貼ったか(`make check` の出力など)。
- アーキテクチャ判断を含むなら、対応する `docs/adr/` のADRへリンクしたか(含まないなら N/A)。

受け入れ基準がそのままPRの検証欄・レビューの合否基準になる。基準が曖昧なままPRを開くと、レビュアー(人間または `.claude/agents/reviewer.md`)が合否を判定できない。分解の質は起票時にほぼ決まる。
