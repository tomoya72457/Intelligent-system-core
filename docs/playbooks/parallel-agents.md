# プレイブック: 並列エージェント運用

このファイルの目的: 複数のコーディングエージェントを同時に走らせる場合の安全な分割・統合手順をまとめる。

> **位置づけ(先に読む)**: 本手法は `docs/governance/tech-radar.md` 上で **Trial**(加重平均 3.40、90日TTL)。ソロ開発の通常規模では過剰になりやすい。**複数エージェントが同一ファイル群に触れかねない規模**(このテンプレート自身の構築のように、レーンをまたぐ大きめの作業)でのみ検討する。まずは単一エージェント+良い道具(`make check`・hooks・Issue分解)を既定にする。

## 原則: 知性は並列化、書き込みは直列化

並列化してよいのは「知性」――読解・分析・下調べ・レビューといった、状態を変えない作業。**書き込み(ファイル編集・コミット・マージ)は直列化する**。同じ状態を複数のwriterが同時に書き換えると、片方のコンテキストに他方の変更が反映されず、統合時に食い違う(Cognition「Don't Build Multi-Agents」の教訓)。

役割で分けたマルチエージェント(planner/coder/reviewer を別プロセスで常時協調させる等)を**既定にはしない**。エージェント連携の失敗の79%は仕様の曖昧さと検証不足に起因し、エージェント数を増やしても解けない(MAST分析)。単一エージェント+明確な受け入れ基準+機械検証が既定で、並列はあくまで規模が要求したときの選択肢。

## 手順

1. **レーンを素集合で切る**: 各エージェントに、互いに重ならない(disjoint)ファイル集合を割り当てる。境界はディレクトリ単位が最も安全。
   - 実例: このテンプレートの構築は WS-A(`.github/`)/ WS-B(`.claude/` ほか)/ WS-C(`docs/`)/ WS-D(ルート+`tools/`)/ WS-E(`profiles/`)という素集合レーンに分けている。共有する契約(Makeターゲット名・ラベル名・予算値)だけを事前に固定し、あとは各レーンが独立して書く。
2. **worktreeで作業ツリーを分離する**: `git worktree add` でレーンごとに別の作業ツリー+別ブランチを用意する。同一チェックアウトを複数エージェントで共有しない(片方の未コミット編集が他方を汚染する)。

   ```bash
   # レーンごとに独立した作業ツリー+ブランチを作る
   git worktree add ../work-github  -b lane/github   # WS-A: .github/ のみ触る
   git worktree add ../work-docs    -b lane/docs      # WS-C: docs/ のみ触る
   # 使い終わったら片付ける
   git worktree remove ../work-github
   ```
3. **各レーンで自己検証する**: 各エージェントは自分のレーン内で完結する検証(構文チェック等)を回す。レーンをまたぐ検証は統合時に行う。
4. **直列に統合する**: 書き込みを1本ずつマージする。各マージの直後に `make check` を実行し、緑を確認してから次を取り込む。並列にマージしない。
5. **コンフリクトは設計シグナルとして扱う**: 統合でコンフリクトが頻発するなら、レーン分割(=モジュール境界)が甘い証拠。ファイルの切り方を見直す。`docs/conventions/architecture.md` の境界原則に立ち返る。

## やらないこと

- 同一ファイルを2レーンに割り当てる。
- 複数エージェントの書き込みを同時にマージする。
- 事前契約(共有インターフェース)を決めずに走らせる。仕様の曖昧さが失敗の主因である以上、境界と契約の合意が先。
