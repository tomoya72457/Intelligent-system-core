# プレイブック: 本番書き込みの安全パターン

このファイルの目的: 本番データ(DB・外部API・課金・公開物)への書き込みを伴う操作を、エージェント駆動でも安全に回すためのパターンをまとめる。`docs/governance/tech-radar.md`「本番書込セーフティ」(Adopt, 加重平均 4.55)の運用正本。

## なぜ必要か

AIコーディングツールが本番DBを消し、しかも「破滅的失敗だった」と自己申告した事故が実際に起きている(Replit、2025)。設定ファイル注入で約4,000台に被害を出した攻撃も観測されている(Clinejection)。エージェントは「それらしく」書き込みを実行してしまうため、**書き込み経路そのものに人間の承認と楽観ロックを構造として埋め込む**。散文の注意書きではなく、workflowとロック機構で強制する。

## パターン1: PRマージ=承認(提案と適用の分離)

書き込みを2つのworkflowに割る。

- **提案(read/plan)workflow**: エージェントが走るのはここまで。現状を読み、差分・実行計画・影響範囲を**成果物(PRやプラン)として出す**だけ。本番には触れない。
- **適用(write)workflow**: 人間が**PRをマージした事実**をトリガーに起動する。マージ操作が承認の意味を持つ。エージェントが自分の判断で本番へ書き込むことはない。

この分離により「誰が承認したか」がGit履歴に残り、承認なしの書き込みが構造的に不可能になる。

## パターン2: concurrencyで直列化

適用workflowには `concurrency` グループを設定し、**同時実行を1本に絞る**。ただし `cancel-in-progress` は使わない――走行中の書き込みを途中でキャンセルすると、中途半端な状態を残す。後続は待たせる(直列化)。この設計思想は `.github/workflows/ci.yml` の concurrency(こちらはCIなので `cancel-in-progress: true` でよい)と対になっており、「検証は最新だけでよい/書き込みは取りこぼさない」という非対称を意識する。

## パターン3: 書き込み直前のhash/version再確認(楽観ロック)

読んだ時点と書く時点の間で、対象が第三者に変更されていないかを確認する。

1. 読み取り時に対象の版(ETag / version / 行のhash)を控える。
2. 書き込み直前に版を**再取得**し、控えた値と一致するか確かめる(HTTPなら `If-Match` 条件付きリクエスト、RFC 9110)。
3. 一致すれば書く。ズレていれば**中止して読み直す**。上書きしない。

「読んだつもりの古い状態」に対する書き込みを止める最小の仕組み。kjseoの本番運用でこのパターンの有効性が実証されている。

```
version = read_target()          # 読み取り時の版を控える
plan    = compute_change(...)    # ここまでがエージェントの仕事(提案)
# --- 適用直前 ---
if read_target_version() != version:   # 書く直前に再取得
    abort("対象が変更されている。読み直す。")   # 上書きしない
write_with_if_match(plan, version)     # 版一致を条件に書く
```

## パターン4: ロールバック層(3段)

書き込みが誤っても戻せるよう、粒度の異なる3層を用意する。

1. **kill switch**: 書き込み経路を即時停止するフラグ。異常検知したらまずこれ。
2. **リビジョン**: 直前の正しい版へ戻す(アプリ側の履歴・世代管理)。
3. **PITR(Point-In-Time Recovery)**: DB側の任意時点復旧。最後の砦。

上ほど速く局所的、下ほど遅く網羅的。事故対応は上から順に試す。

## チェックリスト

本番書き込みを含む変更をレビューするとき:

- [ ] エージェントは提案までで、適用は人間のマージがトリガーか。
- [ ] 適用workflowに concurrency 直列化があるか(`cancel-in-progress` していないか)。
- [ ] 書き込み直前に版(ETag/version/hash)を再確認しているか。
- [ ] kill switch とロールバック手段が存在するか。
