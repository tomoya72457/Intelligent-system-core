# handoff: セッション引き継ぎ書

長い作業を中断する時・別セッションや別エージェントに渡す時に、文脈を失わず再開できる引き継ぎ書を残す。会話の全文を貼るのではなく、**次の一手に必要な最小限**を構造化して書く。

## 手順
1. `docs/handoffs/YYYY-MM-DD-<topic>.md` を新規作成する。
2. 次の**7セクション**を順に埋める(空でも見出しは残し「なし」と書く):
   1. **目的**: 何を達成しようとしているか(1-3行)。
   2. **現状**: どこまで進んだか。触ったファイル・ブランチ名。
   3. **決定事項**: このセッションで確定した判断(理由付き)。重要なものは ADR へ。
   4. **未決**: 保留・要確認・詰まっている点。
   5. **次の一手**: 再開時に**最初にやること**を具体的に(ファイル名・コマンドまで)。
   6. **検証コマンド**: 状態を確認する手順(`make check` や個別テストなど、そのまま実行できる形)。
   7. **参照**: 関連する Issue / PR / ADR / spec / ファイルパス。
3. 冒頭に**賞味期限を明記する**: 「この引き継ぎは作成から**2週間**で陳腐化しうる。以後はコードと最新の台帳を正とする」。
4. 書き終えたら**再開者の目で読み直す**: この文書と参照先だけで、会話履歴なしに「次の一手」を実行できるか。できなければ足りない文脈(ブランチ名・前提・落とし穴)を足す。

## 書かないこと
- 秘密の実値(トークン・パスワード・APIキー)。参照が必要なら「.env の◯◯」と変数名で指す。
- 会話ログの全文貼り付け(要点だけを構造化する。全文は文脈を薄める)。
- 未検証の「たぶん動く」。動作確認していない事項は「未決」か「次の一手」に置く。

## 注意
- 引き継ぎ書は一時文書。恒久的な決定は ADR、恒久的な規約は docs/conventions へ書き、ここに埋もれさせない。
- 古い handoff は「次の一手」が消化されたら役目を終える。放置された古い引き継ぎは削除してよい。
- 推測や願望を「現状」に混ぜない。事実(実行結果・コミット)と判断を分けて書く。
