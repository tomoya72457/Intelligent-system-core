# intake: 既定変更を審査して記録する

新ツール・新規約・新ワークフロー・新エージェント機能を**既定に入れるか**を、勘ではなくルーブリックで判定し、記録に残す。日常の実装判断はこのスキルの対象外(その場で実装してよい)。

## いつ使うか
- 「このライブラリ/フレームワークを標準にすべきか」
- 「この規約(命名・分割・レビュー方針など)を全体の既定にするか」
- 「この CI ステップ / hook / エージェント機能を恒久装備するか」

## 手順
1. **審査を委譲する**: `adoption-judge` サブエージェントを起動する。渡す情報:
   - 提案の内容(何を既定に入れるか)と動機(何の問題を解くか)
   - 可逆性の見立て(撤去に何が必要か)
   - 影響範囲(3プロファイルのどれに効くか、変更されるファイル)
   - 知っている根拠(記事・実測・事例。無ければ「無し」と明示し judge に調査させる)
   ルーブリックの正本は `docs/governance/rubric.md`、根拠台帳は `docs/governance/tech-radar.md`。
2. **矛盾チェックの確認**: adoption-judge が `docs/governance/intake/` と `tech-radar.md` を検索し、過去の判定(特に Reject 済みの蒸し返し)と矛盾しないか点検した結果を確認する。
3. **判定記録を作成する**: 出力に従い `docs/governance/intake/YYYY-MM-DD-<slug>.md` を新規作成する。
   型は `docs/governance/intake/TEMPLATE.md`(提案/可逆性/賛成論/反対論/採点表/判定/条件/再審査日)。
4. **tech-radar を更新する**: `docs/governance/tech-radar.md` の該当リング(Adopt/Trial/Assess/Reject)の表に1行追加する。台帳は**追記**し、全文をLLMで再生成しない(context collapse 防止)。
5. **INDEX を更新する**: ADR を伴う判定なら `docs/adr/INDEX.md` にも反映する(判定が設計決定を含む場合)。

## 判定後の扱い
- **Adopt**: 既定として実装に着手してよい。12ヶ月 TTL で再審査対象。
- **Trial**: 同梱するがオプトイン。TTL90日。実運用実証なしに Adopt へ昇格させない。
- **Assess**: 記録のみ。同梱しない。
- **Reject**: 実装しない。理由を記録し蒸し返しを防ぐ。

## 注意
- 設定ファイル・hooks・CI の変更を伴う採用は、それ自体がセキュリティレビュー(ADR 必須)の対象。`adr` スキルへ。
- 判定を経ずに既定を変えない(設計原則: 判定を経ない既定変更禁止)。
