# spec: スペック駆動フロー(サイズ閾値ゲート付き)

大きめの機能は、いきなり実装せず「要件→設計→タスク」の短い仕様を先に作る。ただし**全タスクに適用すると過剰**なので、サイズ閾値ゲートで発火を絞る(このフローは Trial 扱い=オプトイン)。

## サイズ閾値ゲート(まずここで判断)
次のいずれかに当てはまる時だけスペックを作る。**当てはまらなければスペックを書かず直接実装する。**
- 変更が**3ファイルを超える**見込み。
- **新規サブシステム**の追加(新しい責務の固まり)。
- **外部API契約の変更**(公開インターフェース・データ形式・他システムとの取り決め)。

小さな修正・局所的な変更・自明な追加は対象外。ゲートを通らないのに仕様を書き始めない。

## 手順(ゲートを通った場合)
`docs/specs/<feature>/` に3ファイルを生成する。

1. **requirements.md**: EARS 風の受け入れ基準で書く。各基準は**検証可能**な形(何が起きれば満たされたと言えるか)にする。4つの型を使い分ける:
   - 常時: 「システムは常に <性質> を満たすこと」(例: 保存データは常に UTF-8)
   - イベント駆動: 「<出来事> の時、システムは <応答> すること」(例: 不正な入力を受けた時、400 とエラー理由を返す)
   - 状態駆動: 「<状態> の間、システムは <振る舞い> すること」(例: 未認証の間、書込 API を拒否する)
   - 望ましくない挙動: 「<異常> の場合でも、システムは <安全側動作> すること」(例: 外部 API 障害時でも既存データを壊さない)
2. **design.md**: 方針・境界・主要インターフェース・影響範囲・代替案。アーキテクチャ判断を含むなら `adr` スキルで ADR を別途起こす。
3. **tasks.md**: 実装を**独立に検証可能な小単位**へ分解する。各タスク = 1論理変更・目安 ±400行以内・`make check` で確認できる完了条件付き。

## 承認と実装
- 3ファイルが揃ったら**人間の承認を得てから**実装に入る(仕様への合意が先)。承認前にコードを書き始めない。
- 実装は tasks.md の単位で進め、各タスク完了時に `make check` と受け入れ基準で検証する。
- 仕様と実装がずれたら仕様側を先に直す(実装が正、ではなく合意が正)。
- スペック一式は実装後も残す(受け入れ基準は回帰確認とレビューの参照点になる)。

## 注意
- ゲート判断に迷う場合(2ファイルだが外部契約に近い等)は人間に一言確認する方が安い。
- requirements に実装方法を書かない(How は design.md へ)。tasks に新要件を足さない(要件の追加は requirements へ戻る)。
