# skills/ — スキル採用基準

スキルは「必要な時だけ読み込まれる再利用手順」。常時コンテキスト(AGENTS.md)を太らせずに知識を段階開示するための層である。ただし増やすほど選択コストと腐敗リスクが増える。**追加する前に以下の8項目を全て満たすか確認する**(personal-os C-1..C-7 を改良した基準)。

## 採用基準(8項目)

| # | 基準 | 確認する問い |
|---|---|---|
| S-1 | 削除テスト | このスキルを消したら、エージェントは実際にミスをするか?(消しても困らないなら追加しない) |
| S-2 | 頻度 | 月1回以上使う見込みがあるか?(年数回なら docs/playbooks の手順書で足りる) |
| S-3 | 多段ゲート | 「いつ発火すべきでないか」が description と本文のゲートで明確か?(過剰発火は害) |
| S-4 | トリガー description | frontmatter の description だけ読んで、発火すべき場面が第三者に判定できるか? |
| S-5 | 単一責務 | 1スキル=1つの仕事か?(2つ以上の仕事は分割するか、片方を捨てる) |
| S-6 | 150行以内 | SKILL.md 本文が150行以内か?(超えるなら知識の正本を docs/ へ逃がしルータ化する) |
| S-7 | 安定性 | 手順が月次で変わるようなら時期尚早。安定してから収載する |
| S-8 | 定期棚卸 | 月次棚卸で「最終使用日」を確認し、3ヶ月未使用なら削除候補にする運用に耐えるか |

## 設計原則

- **スキルはルータ**: 知識の正本は docs/(rubric, TEMPLATE, conventions)に置き、スキルは「いつ・何を・どの順で」だけ書く。同じ内容を二重に書かない(複製は必ず腐る)。
- **新スキルの既定化は intake を経る**: スキル追加も「既定変更」。`intake` スキル → adoption-judge の判定記録を残す。
- **frontmatter**: `name`(ディレクトリ名と一致)+ `description`(いつ使うかが分かる日本語のトリガー文言)。

## 同梱スキル一覧

| スキル | 用途 | 扱い |
|---|---|---|
| intake | 既定変更の審査(adoption-judge 起動→記録) | Adopt |
| adr | 軽量 ADR の作成 | Adopt |
| spec | サイズ閾値付きスペック駆動フロー | Trial(オプトイン) |
| handoff | セッション引き継ぎ書の作成 | Adopt |
| incident | 事故記録と再発防止の機械化 | Adopt |

## 不採用の記録(蒸し返し防止)

| 提案 | 不採用理由 | 日付 |
|---|---|---|
| commit-message スキル | `.gitmessage` テンプレート+規約1行で足りる(S-1 削除テスト不合格) | 2026-07-09 |
| デイリーレポート生成スキル | 頻度はあるが単なる出力整形で、ミス防止に寄与しない(S-1 不合格) | 2026-07-09 |
| リファクタリング手順スキル | 「いつ発火すべきか」を定義できず過剰発火が確実(S-3 不合格) | 2026-07-09 |
