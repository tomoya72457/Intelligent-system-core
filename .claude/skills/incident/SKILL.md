# incident: 事故記録と再発防止の機械化

事故は責める対象ではなく、**仕組みを1段強くする**機会。観測事実を残し、再発防止を可能な限り「人が気をつける」ではなく**機械的な強制**まで落とす(kjseo 方式: incident → rule → code)。

## 手順
1. `docs/incidents/YYYY-MM-DD-<slug>.md` を新規作成し、次を書く:
   1. **観測事実**: 何が起きたか。時刻・コマンド・エラー・影響範囲。推測と分けて事実だけ。
   2. **原因**: なぜ起きたか(直接原因と、それを許した仕組み上の欠落)。
   3. **対応**: どう収束させたか(復旧手順・ロールバック層: kill switch → リビジョン → PITR)。
   4. **再発防止**: 次項の通り、機械化まで落とす。
2. **再発防止を機械化する(最重要)**。「気をつける」で終わらせず、次の順で**上ほど優先**して実装できないか検討する:
   - **hooks で遮断**できるか → `.claude/hooks/` の該当スクリプト(危険コマンド・秘密読取・設定保護)に条件を追加する提案。ただし hooks 自体の編集は人間が行い ADR に記録(自己改変禁止)。
   - **check_structure.py で検出**できるか → `tools/check_structure.py` にチェックを追加する(構造・参照・予算の逸脱)。
   - **CI で落とす**できるか → `.github/workflows/ci.yml` のゲートに追加する。
   - どうしても機械化できない残差のみ、規約(docs/conventions)や AGENTS.md の1行として明文化する。
3. 機械化の変更が設定/hooks/CI に及ぶ場合は `adr` スキルで ADR を起こし、`incident → rule → code` のトレースを残す。

## トレースの例(incident → rule → code)
- 事故: エージェントが `git add -A` で無関係の一時ファイルをコミットに巻き込んだ。
- rule: 「一括ステージング禁止・パス明示」を AGENTS.md の Never に追記。
- code: `.claude/hooks/block-dangerous.sh` に `git add -A/.` の遮断条件を追加(人間が編集し ADR 記録)。
- 記録: `docs/incidents/` の記録に、この rule と code の該当行へのリンクを書き、追跡可能にする。

## 注意
- 「二度と手で気をつける」は再発防止として弱い。1件目は記録、2件目は必ず仕組み化(ミス2回で追記/機械化の原則)。
- 秘密露出インシデントは、露出値の無効化(ローテーション)を対応に必ず含める。記録には実値を書かない。
- ヒヤリハット(実害なし)も記録対象。実害が出る前の機械化が最も安い。
