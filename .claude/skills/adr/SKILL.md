# adr: 軽量アーキテクチャ決定記録

「なぜこの設計/選定にしたか」を、重い RFC ではなく軽量な ADR で残す。比例原則: 小さな実装判断には書かない。**アーキテクチャ判断・不可逆な選定・設定/hooks/CI の変更**の時に書く。

## いつ書くか(閾値)
- 技術・ライブラリ・ツールの選定(代替を退けた判断)。
- モジュール境界・レイヤ構成・依存方向の決定や変更。
- `.claude/` `.gemini/` `.github/workflows/` などエージェント設定・CI の変更(セキュリティ境界。必須)。
- 後で読む人が「なぜ?」と問う可能性が高い方針。

## 手順
1. **採番する**: `docs/adr/INDEX.md` を見て次の連番(4桁ゼロ埋め、例 `0006`)を決める。
2. **本体を作成する**: `docs/adr/TEMPLATE.md` をコピーして `docs/adr/NNNN-<slug>.md` を作る(slug は英小文字ハイフン区切り、例 `0006-adopt-import-linter.md`)。
   記入する欄: タイトル / 日付 / 状態 / 文脈(上限20行) / 決定 / **検討した代替案(必須)** / 結果 / Confirmation / 信頼度と再検討トリガー。
   - 代替案には「なぜ退けたか」を各1-2行で必ず書く。代替案が挙がらない決定は、まだ決定の体を成していない。
   - 再検討トリガーは具体的に書く(例: 「管理リポジトリが5を超えたら」「◯◯が個人アカウントで使えるようになったら」)。
3. **Confirmation 欄は実装後に追記する**: 「どう検証されたか」は決定時点では空でよい。実装・検証が済んだら、どのテスト/`make check`/挙動で裏付けたかを追記する。
4. **INDEX を更新する**: `docs/adr/INDEX.md` の一覧表に1行(番号/タイトル/状態/日付)を追加する。台帳は追記し、全文再生成しない。

## 状態の遷移
- `Proposed`(起案)→ `Accepted`(合意・実装可)→ 必要に応じて `Superseded by NNNN`(置き換え)/ `Deprecated`(廃止)。
- 状態が変わったら ADR 本体と INDEX の両方を更新する。古い ADR は**消さず**、置き換え先番号を記す(履歴が判断の資産)。

## 注意
- 文脈は20行以内に絞る(背景の垂れ流しをしない)。決定と代替案を主役にする。
- 既定変更(新ツール採用など)を伴う場合は先に `intake` スキルで審査し、その判定記録へ ADR からリンクする。
- 1 ADR = 1 決定。複数の決定を1本に詰め込まない(後から個別に覆せなくなる)。
